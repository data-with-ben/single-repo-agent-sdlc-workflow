## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
1. Choose and add persistence stack: SQLAlchemy 2.0 (ORM) plus Alembic (migrations) as backend dependencies, with SQLite as the default database (file-based, zero-config, matches the "runs clean from an empty database" AC and the later seed-script task). DATABASE_URL is env-configurable with a SQLite file default so a future task can swap engines without code changes.
2. Define ORM models for every entity in spec.md Section 4: User, Client, Assignment, TimeEntry, Season, Team, Game, ObjectiveResult, Holding, Transaction, Dividend, Wallet -- fields exactly as listed there. Integer autoincrement primary keys throughout (spec only says "id", type is an implementer choice). Verifies AC #1.
3. Model foreign key relationships explicitly: Assignment -> User (consultantId) and Client (clientId); TimeEntry -> User and Client; Team -> Season, and Team membership -> User; Game -> Season and two Teams (home/away); ObjectiveResult -> Game/User/Team; Holding/Transaction/Dividend/Wallet -> User and consultant (User). Enable SQLite foreign key enforcement explicitly via a PRAGMA on connect (off by default in SQLite). Verifies AC #4.
4. Set up Alembic under backend/ (alembic init, configure env.py to import the models' metadata and read DATABASE_URL), then generate one initial autogenerated migration covering all entities/FKs from step 2-3. Verifies AC #2.
5. Documented ER diagram: hand-written Mermaid erDiagram in backend/docs/er-diagram.md reflecting every entity and relationship from steps 2-3 (renders natively on GitHub, no extra diagram-generation dependency needed). Verifies AC #3.
6. Tests: a pytest fixture spins up a fresh SQLite file/in-memory DB and runs the Alembic migration (or Base.metadata.create_all) against it; tests assert (a) migration runs clean from empty, (b) every entity/table exists with its spec'd columns, (c) a foreign-key violation (e.g. TimeEntry referencing a nonexistent User) raises an IntegrityError. Verifies AC #2 and #4 concretely.
7. Update backend/README.md with migration commands (alembic upgrade head, alembic revision --autogenerate) and a pointer to the ER diagram doc.
8. Verification pass: run the new migration against a clean empty DB, run the new pytest suite, and visually confirm the Mermaid diagram's entities/relationships match the models before handing off to unit-tests.

What was implemented:

- Defined all 12 entities from SPEC.md Section 4 as ORM models in app/models.py: User, Client, Assignment, TimeEntry, Season, Team, Game, ObjectiveResult, Holding, Transaction, Dividend, Wallet.

- Added Alembic under backend/alembic, wired env.py to the models' metadata and DATABASE_URL, and generated one initial autogenerated migration covering all 12 tables and their foreign keys.

- Documented the schema as a Mermaid ER diagram in backend/docs/er-diagram.md, including an explicit "deliberate deviations" section.

- Updated backend/README.md with migration commands and a pointer to the ER diagram.

- Added *.db / *.db-journal to .gitignore.

Key technical decisions (resolving hostile-plan-review warnings):

- TeamMembership is a new join table normalizing Team.memberIds[] so membership is foreign-key-enforced against users, rather than an unenforceable array column.

- ObjectiveResult gained a game_id FK (not in the spec's literal field list, which only names gameDate) so the row is foreign-key-enforced against a real Game; game_date is kept alongside it per the spec's naming.

- Wallet uses user_id as its primary key (no synthetic id), matching the spec's (userId, balance) field list -- the only entity without an autoincrement id.

- Both deviations are documented in code docstrings and in the ER diagram's "deliberate deviations" section so they read as intentional, not oversights.

- User.roles is stored as a JSON column (list of strings) rather than a separate table, since it's a small fixed vocabulary (admin|consultant) and SQLite has no native array type.

Integration points:
